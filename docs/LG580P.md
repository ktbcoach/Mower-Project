# LG580P GNSS logger (fork of the Watson logger)

Parallel logger for the **SparkFun LG580P GNSS Flex pHAT** (Quectel LG580P —
dual-antenna RTK GNSS with heading). Lives alongside `watson_dms`; the Watson
logger is preserved and untouched.

## Hardware

| | |
|---|---|
| Receiver | SparkFun LG580P Flex pHAT (dual-antenna RTK + heading, up to 20 Hz) |
| Interface | Pi **primary UART**, GPIO14 (TX) / GPIO15 (RX) → `/dev/serial0` |
| Baud | **460800** (confirmed default; `detect` sweeps 9600–921600) |
| Output | NMEA 0183 (GGA, RMC, VTG, GSA, GSV…) + Quectel **PQTM** (config + `PQTMTAR` heading) |
| RTK status | GGA fix-quality field: 4 = RTK fixed, 5 = RTK float |
| Switch/LEDs | Multi-IO HAT dry-contact + LEDs over I2C (coexists — different pins) |
| RTCM radio | Multi-IO HAT **RS232** (UART5 → `/dev/ttyAMA5`, upper DB9, J2 RX jumper) |

## Setup on the Pi

```bash
sudo bash scripts/setup_pi_lg580p.sh      # enable primary UART (PL011), disable console
sudo reboot
# (watson_dms + lg580p install together)
cd ~/gps-collector && source .venv/bin/activate
pip install -e ".[pi]"
```

## Bring-up

```bash
python -m lg580p detect                    # find the baud rate
python -m lg580p capture --baud 460800 --seconds 5   # see raw sentences + a type summary
```

The `capture` summary lists which sentence types appear (and flags bad
checksums) — use it to confirm the receiver is emitting GGA/RMC/VTG and the
`PQTMTAR` heading message.

## Logging

```bash
# Continuous (writes logs/lg580p-<timestamp>.csv, live status line):
python -m lg580p collect --gpx logs/track.gpx

# Switch-gated field service (Multi-IO HAT dry-contact ch1 gates logging):
python -m lg580p collect --switch

# Auto-start at boot:
sudo bash scripts/install_lg580p_service.sh
journalctl -u lg580p -f
```

LED status (HAT): **GPS LED** off=no fix · blink=fix (GPS/DGPS/RTK-float) ·
solid=**RTK fixed**. **Logging LED** off=idle · blink=logging.

**IMU fusion (50 Hz, coasts through RTK-float):** with an LSM6DSO wired to SPI,
`python -m lg580p fuse` runs an error-state Kalman filter over IMU + GNSS and
logs a smoothed 50 Hz solution. See [EKF.md](EKF.md).

> Only run one service at a time — `lg580p` and `watson-dms` both grab a serial
> port and the HAT. `sudo systemctl disable --now watson-dms` before enabling
> this one.

## Dual-antenna heading setup

Heading is computed from the vector between the two antennas. Preloading the
**fixed baseline distance** (their exact center-to-center separation) constrains
RTK ambiguity resolution so heading resolves faster and more reliably. Stop the
logger/service first (the port must be free), then:

```bash
# measure the antenna separation precisely; e.g. 1.000 m:
python -m lg580p config set-baseline 1.000      # sends PQTMCFGBLD + PQTMSAVEPAR
# then power-cycle the module.

python -m lg580p config get-baseline            # read it back
python -m lg580p config send "PQTMCFGRCVRMODE,W,1" --save   # raw command (rover mode)
```

`config` builds the checksum for you and prints the receiver's `$PQTM…`
acknowledgement. Baseline range 0.000–5.000 m; `0` = auto-calculate.

> If heading still won't solve after setting the baseline, the issue is upstream:
> both antennas must be connected with a clear sky view and a real physical
> separation — a preset baseline can't fix a missing/blocked second antenna.

## RTK corrections (RTCM3)

RTK-fixed needs an RTCM3 correction stream. The LG580P accepts RTCM3 on its UART
RX — which is the Pi's TX on `/dev/serial0` — so the Pi just **writes** the
correction bytes to the same port it reads NMEA from (full-duplex).

Chain for this project:

```
VTrans RTN (NTRIP) → [base Pi: ntrip_to_serial.py] → base radio ))) )))
    ((( rover radio → Pi /dev/ttyUSB0 → lg580p collect --rtcm-source → LG580P
```

**Base Pi** (pulls corrections, feeds the base radio). VTRI single-base station:

```bash
# credentials via env vars (keeps them out of shell history and the repo):
export NTRIP_USER=your_user NTRIP_PASSWORD=your_pass

# list mountpoints (pick an RTCM 3.x one, NOT CMRx — LG580P can't decode CMRx):
python3 tools/ntrip_to_serial.py --host 20.185.11.35 --port 2101 --list

# validate the stream (no radio; prints byte count + RTCM msg types).
# VCAP_RTCM3 = nearest single-base station (no GGA needed):
python3 tools/ntrip_to_serial.py --host 20.185.11.35 --port 2101 --mountpoint VCAP_RTCM3

# then bridge to the base radio (MaxStream XStream @ 19200):
python3 tools/ntrip_to_serial.py --host 20.185.11.35 --port 2101 --mountpoint VCAP_RTCM3 \
  --serial /dev/ttyUSB0 --serial-baud 19200
```

Credentials come from `NTRIP_USER` / `NTRIP_PASSWORD` (or `--user`/`--password`).
Don't commit them — put the `export` lines in a gitignored file you `source`.
Pick an **RTCM 3.x** mountpoint (nearest station, or `VRS_RTCM3` + `--lat/--lon`
for a network solution). CMRx mountpoints are Trimble-proprietary and unusable here.

**Base Pi auto-start (service).** The repo is cloned as `~/Mower-Project` on the
base Pi. To start the bridge on every boot instead of running it by hand, use
`scripts/start_base.sh` (manual launcher) or install the systemd service:

```bash
cd ~/Mower-Project && git pull

# one-off manual run (creds from scripts/ntrip-base.env or the environment):
bash scripts/start_base.sh

# auto-start on boot:
sudo bash scripts/install_base_service.sh    # 1st run drops scripts/ntrip-base.env, then stops
nano scripts/ntrip-base.env                  # fill in NTRIP_USER / NTRIP_PASSWORD
sudo bash scripts/install_base_service.sh    # 2nd run installs + enables

systemctl status base-ntrip                  # manage like the rover's service
journalctl -u base-ntrip -f
```

`scripts/ntrip-base.env` holds the VTrans credentials and is gitignored (never
committed). Defaults match the manual commands above (caster `20.185.11.35:2101`,
mountpoint `VCAP_RTCM3`, radio `/dev/ttyUSB0` @ `19200`); override at install with
env vars, e.g. `sudo MOUNTPOINT=VRS_RTCM3 LAT=44.26 LON=-72.58 bash scripts/install_base_service.sh`.

**Windows base station.** The same bridge runs on a Windows PC — double-click
`scripts\start_base.bat` (a COM port instead of `/dev/ttyUSB0`, same gitignored
`ntrip-base.env`). See [`docs/WINDOWS_BASE.md`](WINDOWS_BASE.md).

**Android base station.** A phone/tablet runs the bridge *and* the rover
dashboard as one Kivy app in Pydroid 3, forwarding corrections out USB-C to the
serial radio (`usbserial4a` in place of pyserial). See
[`android/README.md`](../android/README.md).

**Rover Pi** (injects corrections while logging). The rover radio plugs into the
Multi-IO HAT's **RS232** port (upper DB9), which is UART5 → `/dev/ttyAMA5`:

```bash
# rover radio = MaxStream XStream @ 19200 on the HAT RS232:
python -m lg580p collect --rtcm-source /dev/ttyAMA5 --rtcm-baud 19200 --gpx logs/t.gpx
# service: sudo RTCM_SOURCE=/dev/ttyAMA5 RTCM_BAUD=19200 bash scripts/install_lg580p_service.sh
```

The live status line shows `CORR` when corrections are flowing. As they reach the
receiver, watch `PQTMTAR` quality go 5 (float) → **4 (fixed)** and the baseline
converge to the true antenna separation.

> The radio must present **RS-232 levels** (±12 V) to use the HAT's DB9 — a
> TTL/USB radio would instead go to `/dev/ttyUSB0` or a TTL UART. RX-only is
> enough (corrections flow radio → Pi); set `--rtcm-baud` to the radio's baud.

> A **VRS/network-RTK** mountpoint needs the client to send a GGA position
> (`--lat/--lon`); a **single-base** (nearest CORS station) mountpoint does not.
> Pick a base station close to the mowing area for best results.

> **Auto-tracking the rover (`--gga-from-rover`, on by default in the launchers).**
> Instead of a hand-entered `--lat/--lon`, the bridge can build the VRS GGA from
> the rover's own position, read out of the `$PRSTAT` telemetry it sends back over
> the radio. The virtual base then follows the rover automatically — no coordinate
> to keep in sync. It bootstraps on the rover's **first valid fix** (the standalone
> ~2 m fix reported before corrections arrive — it can't wait for float, or it
> would deadlock: float needs corrections, which need the GGA). A fixed
> `--lat/--lon` still works as a **seed** so corrections start flowing immediately;
> the rover position takes over the moment telemetry arrives. Set
> `GGA_FROM_ROVER=0` in `ntrip-base.env` to force the fixed position instead.

### Stuck at RTK-Float (won't promote to Fixed)

Float means corrections *are* arriving but the receiver can't resolve carrier-phase
integer ambiguities. If signal and geometry look good (high sats, C/N0, low HDOP)
and the dual-antenna **heading** solves to Fixed while **position** stays Float,
the rover is healthy — the problem is the corrections. Two usual causes:

1. **Baseline too long.** A single-base (`VCAP_RTCM3`) station more than ~15–20 km
   away won't Fix — ionospheric error decorrelates. Switch to the **VRS network
   mountpoint** (`NTRIP_MOUNTPOINT=VRS_RTCM3` + `LAT`/`LON` in `ntrip-base.env`);
   the caster synthesizes a virtual base at your position, so the baseline is short.

2. **Correction age.** A slow radio (e.g. XStream @ 19200) may not pass a full RTCM
   epoch within ~1 s; the **age of corrections** then climbs and Fix never happens.
   Watch **Corr age** on the base display (and `age=` on the rover status line):
   ≤2 s is healthy, climbing past ~8 s means the link can't keep up — use a lighter
   mountpoint (fewer constellations / MSM4 not MSM7) or a faster radio.

## Base station display (rover telemetry)

In the field the rover has no Wi-Fi/SSH, so it reports its status **back over the
same radio link** and the base Pi shows it on screen. The correction radios are
transparent serial, so the link is made two-way: base → rover carries RTCM,
rover → base carries a compact `$PRSTAT` status line (~1/s).

```
VTrans RTN → [base: ntrip_to_serial.py] → base radio ))) RTCM ))) rover radio → LG580P
   base display ← rover-status.txt ← [base: ntrip_to_serial.py] ((( $PRSTAT ((( rover: lg580p --telemetry
```

**Wiring (one-time):** the rover must now *transmit*. Wire the HAT RS232 **TX
(DOUT) → radio DIN**, and confirm the base radio's **DOUT → USB-serial RX**. The
XStreams share one half-duplex channel, so telemetry is kept to one short line
per second to avoid starving corrections.

**Rover** — add `--telemetry` to `collect` (or `TELEMETRY=1` on the service). It
sends `$PRSTAT` out the RTCM radio; needs `--rtcm-source` set:

```bash
python -m lg580p collect --switch \
  --rtcm-source /dev/ttyAMA5 --rtcm-baud 19200 --telemetry
# service (autoboot, switch/LED logging + telemetry):
sudo RTCM_SOURCE=/dev/ttyAMA5 RTCM_BAUD=19200 TELEMETRY=1 \
     bash scripts/install_lg580p_service.sh
```

`$PRSTAT` carries: fix quality, sats used/tracked, **C/N0 max/avg** (antenna
health at a glance), lat/lon/alt, heading + quality, HDOP, speed, and
logging/corrections flags. C/N0 comes from a new GSV aggregation in the assembler.

**Base** — the bridge mirrors the latest line to `rover-status.txt`
(`--status-file`, on by default in the service), and a Tkinter dashboard reads
it:

```bash
sudo apt install -y python3-tk                 # once
bash scripts/install_base_display.sh           # desktop launcher "Rover Status"
# or run directly:
python3 tools/rover_display.py                 # add --fullscreen for kiosk
```

The dashboard shows a big colour-coded fix-state banner, sats + signal,
position/heading, and a **LINK LOST** flag if telemetry goes stale (>5 s). The
display only reads the file (no serial), so it coexists with the base bridge that
owns the radio port.

## Notes

- **Heading** comes from `PQTMTAR` field 7 (0–360°); the receiver's `$GNTHS` can
  stay void even while `PQTMTAR` reports heading, so `PQTMTAR` is the primary
  source. `PQTMTAR` order (confirmed): `MsgVer,UTC,Quality,Res,Length,Pitch,
  Roll,Heading,AccPitch,AccRoll,AccHeading,UsedSV`. Quality 4 = RTK fixed,
  5 = RTK float. A 2-antenna system leaves **Roll** empty (heading + pitch only).
- `GnssReading` reuses `latitude_deg` / `longitude_deg` / `altitude_m` /
  `has_gps_fix`, so a GPX writer can log it directly.
